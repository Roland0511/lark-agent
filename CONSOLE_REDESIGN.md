# 控制台重设计决策记录

> 状态：已实现并验收。六个一级页面、两组关键二级体验及安全恢复方案 B 已按确认方案落地，并完成桌面、平板、移动端与交互回测。

## 已确认方向

### 运行总览

- 整体信息架构采用“运行对象”方向：先给系统结论，再按机器人、Runner 等运行对象分层展示健康状态。
- “最近任务”区域采用行动优先方向中的列表设计：任务名称、状态、执行器、更新时间为首层信息，点击后再查看完整链路。
- 参考图：
  - `artifacts/console-redesign/overview-object-health-reference.png`
  - `artifacts/console-redesign/overview-recent-tasks-reference.png`

### 任务中心

- 合并现有“处理流水”和“任务中心”为一个一级页面“任务中心”。
- 页面采用任务流态势方向：以正在处理的任务为主体，展开任务时展示八阶段进度；筛选、收件箱、发件箱、诊断、时间线和原始 Trace 作为分层下钻能力保留。
- 不删除现有任务操作、过滤条件、诊断信息或任务详情能力。
- 任务详情采用八阶段流程画布方向：异常阶段自动展开为“进入条件 → 安全核验 → 处理结果”，正常阶段保持收起；聊天记忆阻塞时隐藏无效重试，只引导进入聊天记忆恢复。
- 任务详情首层分别展示任务实际 Thread 与聊天长期 Thread；阶段耗时、链路诊断、时间线和原始 Trace 依次下沉，原始 Trace 默认折叠在最深层。
- 参考图：
  - `artifacts/console-redesign/task-center-flow-reference.png`
  - `artifacts/console-redesign/task-detail-stage-canvas-reference.png`

### 机器人

- 采用主从分栏结构：左侧是机器人列表，右侧聚焦当前机器人。
- 当前机器人先显示接入结论，并把应用权限、消息订阅、凭据、主人绑定、启用状态和系统通知身份作为独立状态展示。
- 角色与路由、群聊与主人、接入与安全使用折叠分组承载；危险操作继续收在深层配置或更多菜单中。
- 工程复审新增“聊天记忆”二级体验：不增加一级导航，入口并入当前机器人详情；机器人首层只显示聊天总数、阻塞数和最近活动，再按需进入聊天记忆列表与详情。
- 聊天记忆列表至少区分“已绑定 / 已阻塞 / 待初始化”，首层优先呈现机器人、聊天、状态、最后活动；Chat ID、Thread、固定执行环境、工作区和自动压缩记录放入详情。
- 聊天记忆详情需分别确认正常态和阻塞态；阻塞态不能暗示普通任务重试能够解除长期绑定。
- 已确认采用恢复方案 B：阻塞态提供“检测并恢复”，但仅在原执行器环境、Profile、工作区和环境指纹全部通过后台验证后解除阻塞；验证失败时保持阻塞并明确指出不一致项，不把普通任务重试当作恢复入口。
- 聊天记忆采用二级工作台方向：从机器人详情进入，保持“机器人”一级导航选中；左侧低密度浏览聊天，右侧先给阻塞结论，再依次展示恢复条件、关联任务入口、技术绑定信息和自动压缩记录。
- 参考图：
  - `artifacts/console-redesign/bots-master-detail-reference.png`
  - `artifacts/console-redesign/bots-chat-memory-secondary-workspace-reference.png`

### 执行器

- 采用主从分栏结构：左侧是执行器列表，右侧聚焦当前执行器。
- 首层分别展示在线状态、领取任务模式、设备凭据和活跃任务/容量，不把这些状态合并为单一“健康”结论。
- 环境与能力、设备端管理、安全与生命周期使用折叠分组承载；设备端管理仍只提供本机命令复制，不暗示控制台能够远程操作设备。
- 更新与维护操作放在页头附近；撤销凭据、删除等危险操作收在更多菜单和二次确认之后。删除仍需满足“已停用且没有未结束任务”的现有门禁。
- “环境与能力”补充永久聊天记忆能力，以面向用户的语义展示，不直接把 `chat_context_v1` 作为主文案；进入维护、停用、撤销凭据和删除的确认文案需说明已有聊天的固定绑定可能转为阻塞。
- 当前只读 API 没有返回某执行器绑定的聊天记忆数量，因此设计不新增“已绑定聊天数”等无法由现有数据支持的指标。
- 参考图：`artifacts/console-redesign/workers-master-detail-reference.png`

### 待处理

- 采用行动收件箱式主从分栏：左侧集中处理审批决定与发送结果确认，右侧聚焦当前事项的语义、上下文与安全处置。
- “待审批”和“发送结果不确定”保持两类独立语义；列表首层只显示类型、事项、任务、时间与到期信息，技术细节在右侧按需展开。
- 审批详情明确说明“拒绝仅阻止本次动作，任务随后继续执行”，不把拒绝误导为取消任务。
- 发送结果不确定项只承载普通发件箱 `unknown`；保留“原键重试、标记已发送、放弃”三种现有动作，并明确原键重试复用原幂等键。
- 参考图：`artifacts/console-redesign/pending-master-detail-reference.png`

### 故障中心

- 采用单列分层故障台账：首层先给当前未恢复数量和严重程度，再按紧急度排列故障，单条按需原位展开；已恢复历史降级到下方次要区域。
- 严重程度与生命周期保持独立：严重程度为“严重/警告”，生命周期为“待处理/已确认/已恢复”。
- “确认已知”只把故障标记为已确认，不代表已经解决；底层检测恢复后才由系统自动关闭。
- 不新增手动解决、关闭、忽略、重开、分派、静默或修复动作。
- 参考图：`artifacts/console-redesign/incidents-layered-register-reference.png`

## 全局信息规则

- 首屏只显示结论、需要介入的事项和当前工作；技术明细通过展开、详情页或配置弹窗获取。
- 先使用间距、分组、对齐和文字层级，再使用分隔线；避免卡片套卡片和指标卡墙。
- 飞书应用权限、凭据状态、启用意图、主人绑定、系统机器人身份、消息事件订阅必须保持独立语义。
- Runner 在线状态、领取任务模式、设备凭据、活跃任务必须保持独立语义。
- `queued`、`waiting_worker`、`waiting_input`、任务状态与会话状态不得为了简化而合并成错误结论。
- 保留当前深蓝侧栏、钴蓝主色、暖白内容面、中文运维语气和现有图标体系。

## 实现与验收状态

原定六个左侧一级页面和工程复审新增的两组二级体验均已完成设计确认、实现和验收：

- 一级导航固定为运行总览、任务中心、机器人、执行器、待处理、故障中心；原“处理流水”已并入任务中心，不再作为独立一级页面。
- 任务详情按八阶段流程分层展示；机器人详情内提供聊天记忆二级工作台，均不新增一级导航。
- 原有任务、审批、发件箱、执行器与故障生命周期能力保留；仅调整信息架构、视觉层级和自然下钻方式。
- 安全恢复方案 B 已完成失败、成功和“恢复不自动重试任务”三类验证。

以下是已确认页面的小幅规则补充，不要求重新生成整页方案：

- 执行器：增加永久聊天记忆能力的人类可读表达，并补充维护、停用、撤销凭据、删除对固定聊天绑定的影响说明。
- 待处理：范围限定为审批决定与发送结果确认，不再使用“所有需要主人介入的事项”这一宽泛表述。
- 故障中心：已选方案 1 保持不变；聊天记忆 `blocked` 不是新的故障类型，不能默认把所有阻塞聊天混入故障台账。

产品边界已确认：在现有只读诊断能力上追加一条安全恢复工作流。恢复入口位于聊天记忆阻塞详情中，由后台核验原执行器环境、Profile、工作区和环境指纹；全部一致后才解除阻塞并允许关联任务重试，未通过时只展示诊断结果并保持阻塞。

完整的信息架构、桌面/平板/移动端截图、方案对比、交互回测、安全恢复审计以及构建和测试结果记录在 `design-qa.md`。
